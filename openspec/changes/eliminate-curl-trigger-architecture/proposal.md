# Replace trigger pods and CronJobs with API service proxy and internal scheduler

## Why

The previous architecture required the MCP server to create ephemeral
`curlimages/curl` trigger pods in-cluster to POST to workflow `/run` endpoints.
CronJob resources were generated by the CLI and managed separately. This created:

- A dependency on `curlimages/curl` in every trigger path.
- An ipset sync race: kube-router populates `podSelector` ipsets via pod IP.
  Ephemeral trigger pods may connect before their IP is in the ipset, causing
  connection failures on first attempt.
- Pod lifecycle overhead (schedule, image pull, container start, cleanup) for
  what is fundamentally a single authenticated HTTP call.
- A separate NetworkPolicy (`{wf}-trigger-egress`) that had to be generated,
  applied, and cleaned up alongside CronJobs.
- A `proxy-prewarm` init container (also `curlimages/curl`) that blocked pod
  readiness while warming the esm.sh module cache.

All of this complexity is eliminated by using the Kubernetes API service proxy
for HTTP calls and an internal cron scheduler for scheduling.

## What Changed (PRs #5-9)

**PR #5 -- API service proxy for wf_run:**
Replaced ephemeral curl pod creation with a direct HTTP call via the K8s API
service proxy (`/api/v1/namespaces/{ns}/services/{svc}:8080/proxy/run`). The
MCP server uses its in-cluster ServiceAccount credentials for auth. No pods are
created or deleted.

**PR #6 -- Internal cron scheduler:**
Added `robfig/cron/v3` internal cron scheduler. On startup and after each
`wf_apply`/`wf_remove`, the scheduler scans Deployments for the
`tentacular.dev/cron-schedule` annotation and registers cron entries. Each entry
calls `wf_run` internally on schedule.

**PR #7 -- Module pre-warm in wf_apply:**
Added background goroutine in `wf_apply` that extracts JSR/npm dependencies from
the workflow ConfigMap and pre-warms the esm.sh module proxy cache via API service
proxy. `wf_apply` returns immediately; warming runs async.

**PR #8 -- Output schema fix:**
Changed `wf_run` output type from `json.RawMessage` to `any` for MCP SDK
compatibility. `json.RawMessage` was double-encoded by the SDK, producing a base64
blob instead of structured JSON.

**PR #9 -- Error handling for proxy responses:**
Fixed `wf_run` to capture and return 500 responses from the workflow pod (e.g.,
workflow execution errors). Previously, proxied HTTP 500s were silently dropped
and the tool returned an empty/misleading result.

## Net Result

No more ephemeral pods for any trigger path. No more `curlimages/curl` in any
MCP-generated or MCP-triggered workflow. The MCP server is the sole owner of
all trigger, scheduling, and pre-warm behavior.
